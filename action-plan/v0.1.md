## Changes in This Iteration

### 1. Own Parent Menu — "Store Kit"

Replaced the shared JetixWP parent menu with a standalone **Store Kit** top-level admin menu (`dashicons-store`, position 58 — after WooCommerce, before Appearance).

Three submenus:

| Submenu | Slug | React Page |
|---------|------|------------|
| Dashboard | `jwp-stk` | `dashboard` |
| Modules | `jwp-stk-modules` | `modules` |
| Global Settings | `jwp-stk-settings` | `settings` |

All three submenus render the same React app mount point. The PHP localized data passes `currentPage` so React knows which view to render.

**File:** `includes/Admin/class-register.php`

### 2. Dashboard Page — Welcome Banner

The **Dashboard** page now shows:
- A gradient welcome banner with plugin description
- Quick stats (active modules count, available modules count) fetched from the REST API
- CTA buttons linking to Modules and Global Settings pages

The header bar (shared across all pages) displays:
- Left: "Store Kit" title + version badge
- Right: Navigation tabs (Dashboard, Modules, Global Settings) — styled as pill links, active tab highlighted in blue

**Files:** `src/dashboard/App.js`, `src/dashboard/components/DashboardPanel.js`

### 3. Modules Page — Card Grid with Settings Links

The **Modules** page shows the module card grid (same as before) but with enhancements:
- Only **enabled** modules appear (see Module Availability below)
- Each active module card shows a **"Settings"** link next to the title
- Clicking "Settings" opens the **Module Settings Page** within the React app (no page reload)

**File:** `src/dashboard/components/ModulesPanel.js`

### 4. Module Settings Page (Single Module View)

When clicking "Settings" on a module card, the app navigates to a single-module view:
- **Back button** ("← Back to Modules") returns to the modules grid
- **Header card**: module title, tier badge, active/inactive toggle, description
- **Tabbed navigation**: currently only a "General" tab with a placeholder message ("Module settings will be available here once this module's features are implemented.")
- Future iterations will add module-specific tabs and settings here

**File:** `src/dashboard/components/ModuleSettingsPage.js`

### 5. Global Settings Page

The settings page is unchanged in functionality but renamed:
- Admin submenu label: **"Global Settings"**
- Page title: "Global Settings — Store Kit"
- Still contains Theme Compatibility Engine selector + save

**Files:** `includes/Admin/class-register.php`, `src/dashboard/components/SettingsPanel.js`

### 6. Module Availability Control — Single-Point Toggle

Added a **single-point code control** for whether each module appears in the Modules UI at all:

```php
// In includes/class-module-manager.php → register_modules()
'quick-view' => array(
    'title'   => __( 'Quick View', 'jwp-stk' ),
    // ...
    'enabled' => true,   // ← flip this to control visibility
    'default' => false,
),
```

**Rules:**
- `enabled => true` — module appears in Modules page, can be activated/deactivated
- `enabled => false` — module is hidden from UI, cannot be toggled via REST API
- **Default state:** all 6 Core modules are `enabled: true`; all Growth and Power modules are `enabled: false`

**New method:** `Module_Manager::get_available_modules()` — returns only enabled modules.

**New filter:** `jwp_stk_available_modules` — allows external code to override module availability.

**REST API enforcement:** `POST /modules/toggle` rejects requests for non-enabled modules with a `jwp_stk_module_not_available` error.

**Files:** `includes/class-module-manager.php`, `includes/Admin/class-rest-api.php`

## How It All Connects

```
WP Admin Menu
  └── Store Kit (parent)
        ├── Dashboard  → React(currentPage=dashboard)  → DashboardPanel
        ├── Modules    → React(currentPage=modules)     → ModulesPanel
        │                                                   └── [Settings click] → ModuleSettingsPage
        └── Global Settings → React(currentPage=settings) → SettingsPanel
```

- Navigation between Dashboard/Modules/Settings = real page load (different WP admin slug)
- Navigation to Module Settings = React state change (no reload, within same page)

|-----|------|-------------|

| `version` | string | Plugin version |
| `currentPage` | string | `dashboard` / `modules` / `settings` |

### Per-Module Architecture

Each module should follow:
```
includes/Modules/{ModuleSlug}/
├── class-module.php          # Bootstrap, hooks into WC
├── class-frontend.php        # Frontend output
├── class-admin.php           # Admin UI / settings
└── assets/                   # Module-specific CSS/JS
```

The module's settings should be saved under `jwp_stk_options['modules'][{slug}]` and surfaced through module-specific REST endpoints registered in `class-admin.php`.
